# Contribution Impact Harness

A CLI tool that aggregates signals from GitHub and Datadog to evaluate whose engineering contributions are most impactful — designed to help managers recognize and reinforce positive contribution culture.

---

## What it does

For a given date range, the harness pulls data from GitHub and Datadog, attributes metric improvements proportionally across contributors, and generates a structured markdown report per contributor. Reports are manager-facing first, then shared with the contributor as part of a review conversation.

### Signal sources

| Signal | Source | What it measures |
|--------|--------|-----------------|
| Outcome metrics | Datadog (custom + log-based metrics) | Did services get faster, more reliable, less error-prone? |
| Code authorship | GitHub (commits, PRs, labels) | Features shipped, bugs fixed, test coverage added |
| Multiplier effect | GitHub (reviews, comments, docs PRs) | Unblocking others, knowledge sharing |
| Reliability context | GitHub + Datadog monitors | Reverted commits, incident-linked deploys (surfaced as context, never as a penalty) |

### Output

One markdown file per contributor: `reports/{name}_{from}_{to}.md`

- **Summary** — 2–3 sentence narrative generated by Claude
- **Impact Highlights** — top 3 signals, also Claude-generated
- **Signal Breakdown** — tables for each of the four signal types
- **Manager Notes** — blank section for annotation before sharing with the contributor

## Architecture

```
harness/
├── harness.config.toml.example   # template config
├── main.py                        # CLI entry point (click)
├── config.py                      # config loader + validation
├── collectors/
│   ├── github.py                  # commits, PRs, reviews, reverts
│   └── datadog.py                 # custom metrics, log-based metrics
├── attribution.py                 # metric delta → proportional author credit
├── aggregators/
│   ├── code_quality.py            # commits, PRs merged, bug fixes, test files
│   ├── multiplier.py              # reviews given, docs PRs, approvals
│   └── reliability.py            # reverts, incident-linked deploys
├── narrative.py                   # Claude API: generates summary + highlights
├── report.py                      # assembles final markdown per contributor
└── requirements.txt
```

## Quick start

One of `[providers.github]` or `[providers.git]` is required. No Datadog credentials needed.

**With GitHub** (full signal coverage including reviews and approvals):

```toml
# harness.config.toml
[providers.github]
token = "ghp_..."
repos = ["org/api", "org/frontend"]
```

**Without a GitHub token** (local git, no PR review data):

```toml
# harness.config.toml
[providers.git]
paths = ["."]   # one or more local repo paths
```

```bash
pip install -r requirements.txt
python -m harness run --from 2024-01-01 --to 2024-03-31
```

Reports include code authorship and git ownership share. With the `github` provider, the Multiplier Effect section (reviews, approvals) is also populated. Outcome metric sections appear once a metrics provider is connected.

---

## Usage

```bash
python -m harness run \
  --from 2024-01-01 \
  --to   2024-03-31 \
  [--contributor alice@example.com] \
  [--output ./reports]
```

## Providers

The harness is provider-based. Each data source is a separate provider declared in config. Add or remove providers by adding or removing their `[providers.<name>]` section — no code changes needed.

| Provider | Status | Signal categories |
|----------|--------|------------------|
| `github` | Required (or `git`) | code_activity, collaboration, reliability |
| `git` | Required (or `github`) | code_activity, reliability |
| `datadog` | Optional | outcome_metrics |
| `linear` | Stub (coming soon) | outcome_metrics |
| `pagerduty` | Stub (coming soon) | reliability |
| `sentry` | Stub (coming soon) | reliability, outcome_metrics |

## Configuration

Copy `harness.config.toml.example` to `harness.config.toml`:

```toml
# Required — use github OR git (not both)

# Option A: GitHub (enables reviews + approvals in reports)
[providers.github]
token = "ghp_..."
repos = ["org/api", "org/frontend"]

# Option B: local git repos, no token needed
# [providers.git]
# paths = ["/path/to/repo1", "/path/to/repo2"]

# Optional — add any combination
# [providers.datadog]
# api_key    = "..."
# app_key    = "..."
# dashboards = ["abc123", "def456"]

[llm]
model   = "anthropic/claude-opus-4-7"
api_key = "sk-ant-..."

# Maps provider service tags to git path prefixes for metric attribution
# [service_map]
# "api-gateway" = "services/api/"
# "frontend"    = "apps/frontend/"

[output]
path = "./reports"
```

## Attribution model

For each Datadog metric that improved during the period:

1. Look up which Datadog service tag is associated (from metric tags)
2. Map service tag → git path prefix via `service_map` in config
3. Find all commits to that path in the date range
4. Distribute credit proportionally: each author's share = their commits / total commits to that path

This proportional result is the default. Managers can override attribution in the **Manager Notes** section before sharing the report.

## Report format

```markdown
# Contribution Report: Alice Chen
**Period:** 2024-01-01 → 2024-03-31

## Summary
{claude-generated 2–3 sentences}

## Impact Highlights
- {bullet 1}
- {bullet 2}
- {bullet 3}

## Signal Breakdown

### Outcome Metrics (Datadog)
| Metric            | Change        | Attribution                                  |
|-------------------|---------------|----------------------------------------------|
| api.p99_latency   | −42ms (↓)     | 34% (8 of 23 commits to api-gateway)         |
| error.rate        | −0.3%         | 21%                                          |

### Code Authorship
- Commits: 47 | PRs merged: 12
- Features: 5 | Bug fixes: 7
- Test coverage: 9 PRs touched test files

### Multiplier Effect
- PRs reviewed: 18
- Review comments: 64
- Documentation: 2 PRs
- Approvals given (unblocking): 14

### Reliability Context
- Reverted commits: 1 (abc1234 — 2024-02-14)
- Incident-linked deploys: 0

## Manager Notes

> *(Fill this in before sharing with contributor)*
```

## Negative signals

Reverted commits and incident-linked deploys are surfaced as context in the **Reliability Context** section. They are never subtracted from an impact score — the manager decides how much weight to place on them.

## LLM narrative

The Summary and Impact Highlights sections are generated by any LLM you configure. Add an `[llm]` section to your config using a [LiteLLM model string](https://docs.litellm.ai/docs/providers):

```toml
# Anthropic Claude
[llm]
model   = "anthropic/claude-opus-4-7"
api_key = "sk-ant-..."

# OpenAI
[llm]
model   = "openai/gpt-4o"
api_key = "sk-..."

# Google Gemini
[llm]
model   = "gemini/gemini-1.5-pro"
api_key = "..."

# Ollama (local, no key needed)
[llm]
model    = "ollama/llama3"
api_base = "http://localhost:11434"
```

If `[llm]` is omitted entirely, reports are generated without a narrative — just the structured signal tables.

---

## Design decisions

- **No normalization by role or seniority** — impact is impact; humans contextualize
- **Custom date range per run** — supports quarterly, annual, and mid-cycle reviews
- **Manager-first flow** — reports are generated for managers, who annotate before sharing
- **LLM is optional** — the harness runs fully without any LLM configured

## Docker environment

This repo includes a Docker sandbox so Claude Code can run with `--dangerously-skip-permissions` safely. The container runs as a non-root user and only has access to explicitly mounted volumes.

```bash
# Build
docker compose build

# Run
docker compose run --rm claude --dangerously-skip-permissions
```

The container includes `git`, `python3`, and `pip`. `/workspace` maps to this folder on the host.
